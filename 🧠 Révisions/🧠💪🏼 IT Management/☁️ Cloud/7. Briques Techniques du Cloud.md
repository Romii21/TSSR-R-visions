### 7.1 Conteneurisation avec Docker

#### Commandes Docker essentielles

```bash
# Télécharger une image
docker pull nginx:latest

# Lister les images locales
docker images

# Lancer un conteneur
docker run -d -p 80:80 nginx:latest

# Lister les conteneurs en cours
docker ps

# Lister tous les conteneurs
docker ps -a

# Arrêter un conteneur
docker stop container_id

# Supprimer un conteneur
docker rm container_id

# Construire une image depuis un Dockerfile
docker build -t mon-app:v1 .

# Pousser une image vers Docker Hub
docker push username/mon-app:v1
```

### 7.3 Infrastructure as Code (IaC)

#### Terraform

**Fichier de configuration (.tf) :**

```hcl
resource "azurerm_virtual_machine" "vm" {
  name                = "vm-prod-web"
  location            = "France Central"
  resource_group_name = azurerm_resource_group.main.name
  vm_size             = "Standard_B2s"
  
  storage_image_reference {
    publisher = "Canonical"
    offer     = "UbuntuServer"
    sku       = "18.04-LTS"
    version   = "latest"
  }
}
```

**Commandes Terraform :**

```bash
# Initialiser le projet
terraform init

# Voir les changements prévus
terraform plan

# Appliquer les changements
terraform apply

# Détruire l'infrastructure
terraform destroy
```

⚠️ **Attention** : La suppression d'une ligne dans le code Terraform détruit la ressource correspondante !

#### Avantages de l'IaC

1. **Reproductibilité** : Déployer 50 VMs identiques en 10 minutes
2. **Versionnement** : Infrastructure versionnée dans Git
3. **Documentation** : Le code documente l'infrastructure
4. **Collaboration** : Revues de code pour l'infrastructure

### 7.4 Architecture réseau Cloud

#### Réseaux virtuels

**Composantes** :

- **VNet** (Azure) / **VPC** (AWS) : Réseau privé virtuel
- **Sous-réseaux** : Segmentation du réseau
    - Sous-réseau public : Accès Internet
    - Sous-réseau privé : BDD, serveurs internes

**Exemple de plage d'adresses :**

```
VNet : 10.0.0.0/16
  ├─ Sous-réseau public : 10.0.1.0/24
  └─ Sous-réseau privé : 10.0.2.0/24
```

#### Règles de pare-feu

**Exemple** :

- Autoriser RDP (port 3389) uniquement depuis l'IP du VPN entreprise
- Autoriser SSH (port 22) uniquement depuis le réseau admin
- Bloquer tous les autres accès

### 7.5 Scalabilité

#### Scalabilité Verticale (Scaling Up/Down)

**Principe** : Augmenter ou réduire les capacités d'un serveur existant (CPU, RAM)

**Avantages** :

- Simplicité de mise en œuvre
- Pas de modification applicative

**Limites** :

- Plafond de capacité matérielle
- Interruption de service (redémarrage nécessaire)

#### Scalabilité Horizontale (Scaling Out/In)

**Principe** : Ajouter ou supprimer des serveurs

**Avantages** :

- Flexibilité accrue
- Meilleure tolérance aux pannes
- Pas de limite théorique

**Limites** :

- Complexité de gestion
- Nécessite une architecture adaptée (stateless)

#### Élasticité

**Définition** : Capacité à ajuster automatiquement les ressources en fonction de la demande

**Exemple** :

- Si utilisation CPU > 80% pendant 5 min → Ajouter 1 serveur
- Si utilisation CPU < 20% pendant 10 min → Retirer 1 serveur

### 7.6 Haute disponibilité et continuité d'activité

#### Zones de disponibilité

**Définition** : Datacenters physiquement séparés dans une même région

**Protection contre** : Incendie, coupure électrique, catastrophe naturelle localisée

**Exemple Azure** : France Central possède 3 zones de disponibilité

#### Régions géographiques

|Fournisseur|Exemples de régions|
|---|---|
|**AWS**|eu-west-3 (Paris), us-east-1 (Virginie)|
|**Azure**|France Central, West Europe, North Europe|
|**Google Cloud**|europe-west9 (Paris), us-central1 (Iowa)|

⚠️ **RGPD** : Les données sensibles doivent rester en Europe

#### PCA/PRA (Plan de Continuité/Reprise d'Activité)

**Indicateurs clés** :

|Indicateur|Signification|Exemple|
|---|---|---|
|**RTO** (Recovery Time Objective)|Temps max acceptable avant restauration|RTO de 4h = service rétabli sous 4h max|
|**RPO** (Recovery Point Objective)|Perte de données acceptable|RPO de 1h = perte de données max de 1h|

**Solutions Cloud** :

- Sauvegarde automatique (snapshots, backups)
- Réplication géographique entre régions
- Bascule automatique (failover)

### 7.7 Load Balancing (Équilibrage de charge)

**Principe** : Distribution du trafic entre plusieurs serveurs

**Avantages** :

- Optimisation de l'utilisation des ressources
- Évite la surcharge d'un serveur
- Haute disponibilité (si un serveur tombe, les autres continuent)

**Services Cloud** :

- **Azure Load Balancer** / **Application Gateway**
- **AWS ELB** (Elastic Load Balancer)
- **Google Cloud Load Balancing**
